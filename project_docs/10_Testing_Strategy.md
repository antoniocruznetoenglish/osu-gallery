# Testing Strategy: osu gallery

> Promoted out of a single table in `03_Technical_Architecture.md` because "we'll test it" without concrete parameters is how untested code ships. This doc defines what "tested" means for this project, concretely enough that an AI agent can self-check against it.

**Last updated:** 2026-07-16
**Version:** 0.1 · **Status:** Draft / Review / Frozen / Deprecated

---

## 1. Testing Philosophy

| Question | Answer |
|---|---|
| Test-first or test-after? | Write the test alongside the implementation, same commit, not after |
| What must always be tested | Core loop logic (parser, database, search), anything touching stored/scraped data |
| What's explicitly OK to leave untested | Pure UI layout with no logic (PySide6 widget positioning), one-off migration scripts |

## 2. Coverage by Layer

| Layer | Framework (must match `03_Technical_Architecture.md` §1) | Coverage target | What "done" looks like |
|---|---|---|---|
| Unit tests | pytest | All business logic in Pattern Parser, Pattern Storage, Search/Filter modules | Every public function has at least one passing test for expected input and one for invalid input |
| Integration tests | pytest + pytest-qt | Database access, import flow (paste → parse → save), search with FTS5 | Each module boundary in `02_System_Design.md` §1 has at least one test crossing it |
| End-to-end tests | pytest-qt | Critical paths from `01_Product.md` §2 Core Loop | The full core loop (import → search → copy) passes end-to-end at least once per milestone |

## 3. Regression Policy

- A bug fix ships with a test that reproduces the bug first (red), then passes once fixed (green). No fix without a regression test, unless explicitly justified in the PR/commit message.
- Tests are not deleted to make a build pass. A failing test is either fixed, or the feature it tests is deliberately removed (with a Decision Log or Feature Log note explaining why).

## 4. What an AI Agent Must Do

Mirrors into `05_AI_Collaboration_Rules.md` Definition of Done:

1. New logic in a covered layer (§2 above) ships with a test in the same change, not "added later."
2. If a test can't reasonably be written (e.g., requires hardware not available), say so explicitly rather than silently skipping it.
3. Never weaken an assertion (e.g., `assert True`, overly broad try/except swallowing failures) just to get a suite green — flag the real failure instead.
4. Run the existing test suite before declaring a feature done — a change that breaks unrelated tests isn't done, it's regressed.

## 5. Manual / Exploratory Testing

For anything not practical to automate yet (e.g., early-stage UI feel, visual preview quality):

| Area | How it's manually verified | How often |
|---|---|---|
| UI layout and responsiveness | Visual inspection, alt+tab workflow test | Per milestone |
| Static preview quality | Verify patterns render correctly, thumbnails clear | Per milestone |
| Search relevance | Test multi-tag queries, verify results match expectations | Per milestone |
| Copy-to-clipboard | Paste into .osu file, verify code is valid | Per milestone |

## 6. Local-Model-Specific Note

If tests are being generated by a local model with a smaller context window, feed it only the relevant module's code + this doc's §2 row for that layer — not the whole test suite. Verify generated tests actually assert something meaningful (a local model under context pressure will sometimes generate a test that runs without ever asserting a real condition).
